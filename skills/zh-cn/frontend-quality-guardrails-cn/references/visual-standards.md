# 视觉规范

创建、重设计或打磨 UI 时使用本参考。目标不是装饰，而是清晰层级、稳定布局、可读内容和有意图的产品界面。

## 视觉层级

- 首屏应明显呈现主要任务。
- 每个界面只保留一个主导标题层级，避免多个元素争抢页面标题角色。
- 优先用间距和分组建立层级，再增加边框、阴影或背景填充。
- 次要元信息要比标题和操作更安静。
- 仪表盘、表格、侧边栏、卡片、弹窗、工具栏里不要使用 hero 级大字。
- 操作层级要清楚：主要、次要、三级、危险、禁用。

## 密度

- 密度匹配产品类型。SaaS/后台工具应紧凑、可扫描、克制；营销页可以更留白。
- 少量业务数据不要用过大的卡片承载。
- 不要用大面积装饰性空白把关键流程推到折叠以下。
- 重复行/卡片结构保持一致，方便快速扫描。
- 高级控件优先渐进披露，不要挤满首屏。

## 色彩

- 优先使用现有 token。
- 限制强调色，一个主强调色加语义色通常足够。
- 状态不要只靠颜色表达，要配文字、图标或形状。
- 在真实背景上检查 muted 文本对比，而不只在白底上看。
- 避免整个界面都是同一色相的浅深变化。
- 危险色只用于危险或破坏性操作。
- 除非产品风格已有或明确是营销/编辑性界面，否则避免渐变。

## 字体

- 建立清晰尺度：页面标题、区块标题、正文、元信息、说明文字。
- 不使用负字间距。
- 不使用跟随视口宽度变化的字号。
- 长正文不要横跨整个桌面宽度。
- 仪表盘、财务值、计时器和可比较数字列使用等宽数字。
- 除非现有产品使用 Title Case，否则优先句式大小写。
- 长字符串不要全大写，不利于扫描和本地化。

## 间距

- 使用项目间距尺度。
- 相关元素靠近，无关组拉开。
- 区块、表单、卡片和表格的垂直边缘要对齐。
- 工具栏控件间距一致。
- 避免多个嵌套容器都带大 padding，容易让内容显得局促且偶然。
- 不要用手调的一次性 margin 解决本可由布局表达的对齐问题。

## 边框、圆角和阴影

- 按组件类型保持圆角一致。
- 避免带有竞争边框/阴影的卡片套卡片。
- 阴影用于表达层级，不用于装饰。
- 高密度数据区域的边框要轻，但足够定义边界。
- hover 边框不应改变尺寸；空闲态可使用透明边框。
- 同一控件组不要混用太多圆角值。

## 图标

- 使用项目图标库。
- 同一工具栏或列表内图标尺寸和描边保持一致。
- 装饰图标使用 `aria-hidden`，纯图标按钮提供可访问标签。
- 陌生图标不要替代清晰文本，除非有 tooltip/label。
- 图标和文字按视觉中心对齐，而不只是数学中心。

## 版面组合

- 页面区块应全宽或自然分组；除非产品模式要求，不要把整页放进悬浮卡片。
- 避免卡片套卡片。
- 相似项比较用 grid，扫描大量记录用列表/表格。
- 小屏下侧栏和筛选器不要挤占主要内容。
- 最大视觉面积应给重要内容，而不是装饰。
- 响应式变化要有明确策略：堆叠、折叠、滚动或隐藏并提供替代路径。

## 交互状态

- 设计 idle、hover、active、focus、disabled、loading、selected、error、warning、success、empty、skeleton 状态。
- 状态变化不要改变组件尺寸。
- 焦点样式要在所有状态背景上可见。
- 禁用原因不明显时，应提供解释。
- selected 状态要强于 hover 状态。
- loading 指示器要靠近受影响内容或操作。

## 动效

- 动效用于解释因果关系，不用于堆装饰。
- 优先使用 transform 和 opacity。
- 业务工具界面动效时长要短。
- 尊重 reduced motion。
- 不要在重复表格行或密集列表上使用大量动画，除非产品模式需要。

## 常见视觉失败

- 紧凑控件里的文字过大。
- 元信息比主内容更抢眼。
- 边框和阴影同时过多。
- 彩色背景上的灰字对比不足。
- 高密度内容居中对齐。
- 按钮高度不一致。
- 图标导致文字基线漂移。
- badge 抢过标题。
- 空状态看起来像错误。
- skeleton 与最终内容尺寸不匹配。
- 卡片 hover transform 导致网格抖动。
- dropdown/popover 和触发器视觉上脱节。

## 完成标准

视觉工作结束前确认：

- 视线会先落到正确的主要内容或操作。
- 相关控件对齐且尺寸一致。
- 不读每个字也能看出文本层级。
- 长文本有明确处理策略。
- 页面在真实密集数据下仍然可用。
- 没有装饰选择伤害可读性、扫描效率或可访问性。
